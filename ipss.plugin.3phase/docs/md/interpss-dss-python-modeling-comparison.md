# InterPSS and DSS-Python Modeling Comparison

Date: 2026-06-12

This note records the detailed modeling correspondence used during the
InterPSS OpenDSS parser, static power-flow, and QSTS comparison work. The
reference side is DSS-Python/DSS-Extensions behavior for the checked OpenDSS
cases. The InterPSS side is the current static three-phase parser, power-flow,
and QSTS path.

The table focuses on modeling choices and differences that affected parity or
performance during development. It intentionally separates static power-flow and
QSTS behavior from dynamic/DStab behavior.

| InterPSS | DSS-Python / OpenDSS | Notes and modeling details |
|---|---|---|
| `OpenDSSDataParser.forStaticNetwork()` creates a static three-phase network for OpenDSS power-flow and QSTS studies. | DSS-Python compiles OpenDSS circuits into the OpenDSS static circuit model. | Static PF and QSTS must stay on the static network path. Dynamic `DStabNetwork3Phase` objects are not part of the accepted OpenDSS PF/QSTS path. |
| `OpenDSSQstsStudyFactory.from(parser)` requires `parser.isStaticNetworkMode()` and rejects dynamic parsers. | DSS-Python QSTS solves against the OpenDSS circuit state, not a dynamic simulation network. | This enforces the migration goal: OpenDSS QSTS uses `OpenDSSStaticDataParser`, `Static3PNetwork`, and static device controls. |
| Fixed-point PF uses a static three-phase Y matrix and caches/reuses same-pattern Y/factorization state when allowed. | OpenDSS solves the static circuit Y matrix, reusing compiled circuit structure across time steps. | InterPSS currently reports `symbolicFactors=1`, low numeric refactor counts, and `fallbackCount=0` on the accepted controlled Ckt24 8760 run. |
| QSTS state application caches load, generator, and storage base states in `QstsStateApplier` and applies profile multipliers to cached device arrays. | OpenDSS applies LoadShape/PV shape multipliers to the existing circuit device state at each time point. | InterPSS avoids rediscovering devices every step. Long-run result storage should remain `NONE` or monitor-only; `FULL` result sampling materializes all bus/device/branch rows each step. |
| Controlled large-feeder QSTS uses `QstsControlMode.STATIC`, `maxControlIterations=100`, RegControl enabled, and CapControl enabled. | DSS-Python uses `controlmode=static`, `maxcontroliter=100`, RegControl enabled, and CapControl enabled. | Disabled-control runs are treated as frozen-state diagnostics only, not acceptance evidence. |
| Ckt24 controlled 8760 benchmark: InterPSS `1.021454 ms/step`, converged, max PF iterations `10`. | Ckt24 controlled 8760 benchmark: DSS-Python `1.146780 ms/step`, converged, max iterations `9`. | InterPSS runs at about `112.3%` of DSS-Python throughput for this accepted static-control benchmark, exceeding the 80% target. |
| Ckt24 controlled row-level parity passes for voltage, load power, and branch power on common keys. | DSS-Python controlled Ckt24 exports are used as the row-level reference. | Current accepted two-step yearly-profile maxima are about `0.0017546525 pu`, `0.000756435 deg`, `0.06397092 kW`, `0.240896534 kvar`, `0.28418337 kW`, and `0.8739569147 kvar`, with zero tolerance failures. |
| IEEE8500 controlled PV-duty one-step parity passes for voltage, load, branch, generator, and capacitor-state rows. | DSS-Python controlled PV-duty IEEE8500 export is the reference. | Current maxima are about `0.00104704199 pu`, `0.0267427413 deg`, `2.87351585 kW`, `2.728537974 kvar`, `0.014988296 kW`, and `0.00406943705 kvar`, with zero tolerance failures. |
| Static branch-power export samples terminal powers from static branch Y blocks and solved bus voltages. | DSS-Python exports terminal powers from each solved OpenDSS element. | InterPSS has explicit source branches and may export additional near-zero rows. Comparisons normalize element keys, terminal/phase labels, and apply small zero thresholds where needed. |
| Bus-voltage sampling uses active phase masks, not unconditional ABC rows. | OpenDSS reports active node voltages according to each bus/node set. | Phase masks are required because many buses are single-phase or two-phase. InterPSS should not calculate/store unused A/B/C rows for current injection or voltage update when the phase is not active. |
| Fixed-point PF now uses phase-masked static current injection and voltage-update logic. | OpenDSS stamps and solves only active conductors for a device. | InterPSS still has a fixed 3x3 bus-frame internally, so inactive-phase stabilization is needed until the solver moves to an OpenDSS-style conductor-indexed primitive matrix. |
| Missing or inactive phase rows are stabilized with targeted zero-diagonal fill and selected anti-float paths. | OpenDSS avoids zero-padded inactive phase rows by using conductor/node indexing. | InterPSS keeps one general missing-phase diagonal stabilizer while the matrix is fixed ABC. Floating-component deactivation is enabled by default. |
| Long-term direction is an OpenDSS-style primitive Y matrix with device conductor-to-bus/phase mappings. | OpenDSS device models stamp primitive Y matrices into the circuit Y matrix. | This removes zero-padded inactive rows, reduces dimension for partial-phase feeders, and better matches OpenDSS one-phase/two-phase devices. |
| Linecode parser reads header-line and continuation-line `rmatrix`, `xmatrix`, and `cmatrix` independently. | OpenDSS accepts these properties on the same line or continuation lines. | Ckt7 and Ckt24 exposed parser gaps where one property on a line hid another property. Independent property extraction fixed service and overhead line mismatches. |
| Triplex/service linecode parsing preserves `units=kft` even when it appears on the same line as `cmatrix=[...]`. | OpenDSS applies the declared unit to the linecode impedance/capacitance data. | Ckt24 service-line impedance was previously scaled as feet-to-mile instead of feet-to-kft. DSS-Python reported `LineCode.2/0_2/0` after reduction as about `R=0.203485710677 ohm/kft`, `X=0.068094029003 ohm/kft`. |
| Repeated `Neutral=<n> Kron=yes` declarations are counted and applied sequentially. | OpenDSS can apply repeated Kron reductions as declarations are processed. | Ckt24 triplex linecodes intentionally repeat `Neutral=2 Kron=yes`; InterPSS now preserves and applies the reduction count before mapping the one-conductor service result to bus phases. |
| Line geometry parsing matches OpenDSS conductor internal resistance and Carson constants for checked Ckt24 geometries. | DSS-Python/OpenDSS computes geometry matrices from conductor data and OpenDSS constants. | Regression coverage compares Ckt24 `OH3P_FR8_N56_OH_477_AAC_OH_477_AAC_ABCN` against DSS-Python for branch `05410_339569OH`. |
| Static transformer parser maps `%rs=(...)` into winding resistance contribution for normal two-winding transformers. | OpenDSS uses `%rs` winding resistance values in transformer series impedance. | Ckt24 `Transformer.step_05410_G2101CD0200` was previously lossless in InterPSS; DSS-Python showed about `6.58 kW` real loss. After the fix, `z_bb` includes nonzero real resistance. |
| Center-tap service transformer explicit Y includes `%noloadloss` and `%imag` inherited from `XfmrCode` or inline transformer data. | OpenDSS includes transformer exciting/no-load admittance in the service-transformer YPrim. | This removed the dominant IEEE8500 service-transformer loss gap: aggregate service-transformer mismatch went from roughly `49 kW` and `122 kvar` to about `+0.2 kW` and `-9.4 kvar`. |
| Service-transformer `%IMag` is treated as the reactive magnetizing admittance component in the accepted Ckt24 static path. | DSS-Python YPrim evidence showed OpenDSS applies `%IMag=0.5` directly as reactive magnetizing admittance, with `%NoLoadLoss` as conductance. | InterPSS previously interpreted `%IMag` as total exciting-current magnitude and used a Pythagorean reduction, undercounting feeder magnetizing kvar. Fixing this closed the controlled Ckt24 branch-Q residual. |
| Normal transformer no-load admittance is mapped to the static transformer shunt path without replacing tap-dependent transformer Y. | OpenDSS includes no-load loss/magnetizing admittance in transformer terminal admittance while preserving tap behavior. | Ckt24 substation transformer `%noloadloss=0.18` was a key branch-flow parity slice before service-transformer `%IMag` cleanup. |
| Source `R1/X1/R0/X0` is parsed into an internal ideal swing bus plus a source Thevenin branch to the OpenDSS `bus1`. | DSS-Python treats the named circuit `bus1` as the terminal after the `Vsource` Thevenin impedance. | Ckt24 defines `R1=0.63 X1=6.72 R0=4.07 X0=15.55`; InterPSS previously pinned `SourceBus` directly as the ideal swing. The fix removed a false depth-0 voltage offset. |
| Source sequence impedance is converted to phase self and mutual terms using `(Z0 + 2*Z1)/3` and `(Z0 - Z1)/3`. | OpenDSS source impedance is represented internally through sequence-to-phase conversion. | This is needed for unbalanced three-phase source voltage/current behavior in Ckt24 and similar feeders. |
| Load parser accepts `bus=`, parenthesized `kVA=(...)`, default wye connection, `xfkVA`, allocation factors, and post-definition `Load.<id>.AllocationFactor=...`. | OpenDSS supports compact and post-definition property edits. | Ckt7/Ckt24 required updates after the initial load declaration. `xfkVA` allocation must affect actual scheduled load size. |
| Missing `kW` with `xfkVA=0` and a power factor falls back to OpenDSS default load behavior unless `kW=0` is explicit. | OpenDSS defaults an omitted `kW` load to a nonzero default but respects explicit zero. | Ckt24 loads such as `Load.440273200` have no explicit `kW`, `xfkVA=0`, and `pf=0.98`; DSS-Python solved them as the OpenDSS default `10 kW` plus derived kvar. |
| OpenDSS load `model=1` is now tagged and solved with OpenDSS low-voltage model-1 interpolation between `Vlowpu` and `Vminpu`. | OpenDSS/DSS-Extensions uses a low-voltage interpolation for model-1 loads, not a simple extrapolation from generator behavior. | IEEE8500 low-voltage model-1 loads were the residual source after other device fixes. The fix follows OpenDSS load behavior: between `Vlowpu=0.5` and `Vminpu`, current scale interpolates from `Vlowpu` to `1/Vminpu`, then power scales by voltage times that current scale. |
| `Vminpu`/`Vlowpu` fallback is applied to loads, not generalized to generators. | OpenDSS generator voltage limits are generator-specific and should not be inferred from load `Vminpu` behavior. | During IEEE8500 PV-duty debugging, generator mismatch had to be checked separately. The final fix did not blindly apply load fallback rules to generators. |
| OpenDSS load `model=4` CVR uses InterPSS OpenDSS model metadata and CVR exponents. | OpenDSS `model=4` uses nominal linear P and quadratic Q behavior with `CVRwatts`/`CVRvars`. | Ckt7 validation required parsing `model=4`, `CVRwatts`, and `CVRvars`, with defaults consistent with OpenDSS behavior. |
| ZIP/OpenDSS model metadata is retained on static load objects where supported. | DSS-Python/OpenDSS evaluates load model behavior from the OpenDSS model number and ZIPV/CVR fields. | Static PF parity depends on preserving the original OpenDSS model intent, not only a generic constant-power/constant-impedance code. |
| QSTS load state store registers `AclfLoad3Phase` entries from both phase-load lists and contributed load lists. | OpenDSS treats circuit loads as element instances regardless of InterPSS collection placement. | This was needed for contributed static loads so load result sampling and QSTS multipliers did not silently skip loads. |
| QSTS generator sampling uses `OpenDSSStaticGenerator.getTerminalPower3Phase(...)` when available. | DSS-Python reports generator terminal powers under solved terminal voltage. | IEEE8500 PV-duty generator rows required terminal-power sampling, not merely scheduled balanced per-phase generator power. |
| Static PVSystem/generator profiles are converted from OpenDSS time-series data into QSTS generator bindings. | DSS-Python applies daily/yearly/duty PV shapes from OpenDSS objects. | `OpenDSSTimeSeriesData` derives PV available-power profiles from irradiance and temperature shapes and passes them into the generic QSTS schedule model. |
| Storage state is represented through static generator/storage state and QSTS storage base state. | OpenDSS storage carries state of charge across time steps. | Focused mini-case coverage verifies SOC carryover and terminal power behavior in the static QSTS path. |
| RegControl metadata is parsed into `RegulatorControlData`; QSTS can run static stepped regulator controls. | DSS-Python/OpenDSS RegControl operates through the OpenDSS control queue/static control passes. | Current acceptance runs enable static regulator controls. Deeper delayed RegControl queue parity remains a separate future slice, not claimed as fully complete. |
| CapControl metadata is parsed into `CapacitorControlData`; static and delayed capacitor behavior has mini-case reference coverage. | DSS-Python/OpenDSS CapControl can act in static or delayed queue modes. | Controlled large-feeder runs keep CapControl enabled. Delayed control queue operation-count behavior is covered by focused reference data. |
| Capacitor result sampling excludes loads that are actually parsed capacitors from normal load-power rows and records capacitor-state rows. | OpenDSS treats capacitors as capacitor elements, not loads, even though InterPSS may represent fixed capacitor admittance through load-like static objects. | This prevents load-power comparisons from being polluted by capacitor kvar rows and aligns capacitor-state comparison with DSS-Python. |
| Branch key export records OpenDSS element class/name and phase terminal mapping from parser metadata. | DSS-Python keys branch powers by element class, element name, terminal, and conductor. | IEEE8500 and Ckt24 comparisons needed normalized phase labels and parser-supplied class/name mapping, especially around reactors, transformers, regulators, and source branches. |
| Source impedance branches are explicit in InterPSS but filtered or normalized in comparison where needed. | DSS-Python exposes the OpenDSS `Vsource` terminal behavior differently from an explicit InterPSS internal swing branch. | Extra InterPSS-only source rows can appear; accepted comparisons focus on common physical OpenDSS keys and small near-zero thresholds. |
| YPrim diagnostics compare InterPSS static physical branch admittance against DSS-Python element YPrim. | DSS-Python exposes OpenDSS element YPrim for device-level QA. | For sampled IEEE8500 lines and the `hvmv_sub` transformer, YPrim matched DSS-Python, moving debugging toward solved currents, loads, controls, and neighboring model differences. |
| KCL/KVL diagnostics are used before changing solver logic. | DSS-Python reference paths provide solved voltages, currents, and element drops for localization. | A clean InterPSS KCL signal means the model is internally consistent; remaining mismatch usually belongs to device modeling or reference alignment, not sparse summation. |
| Result comparison scripts require the `controls_static` tag by default for large-feeder QSTS acceptance files. | DSS-Python reference exporters also use control-mode-specific filenames. | This prevents accidentally comparing controls-off diagnostic exports against controls-on InterPSS runs. |
| `QstsResultSamplingMode.NONE` is used for long 8760 performance measurement. | DSS-Python performance runner measures solving without forcing a huge all-row result object. | Full all-bus/all-branch/all-device storage for 8760 can be many gigabytes. Production QSTS should use monitor-only or streaming result sinks instead of retaining every row between steps. |
| `QstsResultSamplingMode.FULL` is kept for row-level parity/debug exports. | DSS-Python CSV exporters produce full voltage/load/branch/generator/capacitor rows for comparison runs. | `FULL` is useful for 1-step, 2-step, or 24-step parity gates, but should not be the default for production 8760 studies. |
| Current static PF and QSTS code still has some dynamic/DStab classes in legacy tests and dynamic packages. | DSS-Python reference has no dynamic model layer for these static PF/QSTS comparisons. | The acceptance boundary is that OpenDSS static PF/QSTS entry points use static parsers/networks and do not instantiate/use dynamic DStab models. Legacy dynamic tests remain outside this static comparison scope. |

## Current Acceptance Summary

| Case | DSS-Python reference | InterPSS result | Notes |
|---|---|---|---|
| Ckt24 controlled 8760 QSTS performance | `1.146780 ms/step`, converged, controls enabled | `1.021454 ms/step`, converged, controls enabled | InterPSS is about `112.3%` of DSS-Python throughput and passes the 80% target. |
| Ckt24 controlled yearly-profile row parity | Static controls, DSS-Python voltage/load/branch CSVs | Static parser/QSTS voltage/load/branch CSVs | Two-step yearly-profile row-level gate passes on voltage, load power, and branch power. |
| Ckt24 controlled repeated-state row parity | Static controls, 24-step DSS-Python exports | Static parser/QSTS 24-step exports | Voltage/load/branch comparisons pass; full 8760 row export is intentionally avoided because of size. |
| IEEE8500 controlled PV-duty parity | Static controls, one-step DSS-Python exports | Static parser/QSTS one-step exports | Voltage, load, branch, generator, and capacitor-state comparisons pass with zero tolerance failures. |
| Static parser/PF regression suite | DSS-Python mini and feeder references | InterPSS static parser and fixed-point PF | Focused former failures and full `ipss.plugin.3phase` module gate passed after the final model-1 load fix. |

## Practical Guidance

- Use static parser and static network APIs for OpenDSS PF/QSTS work.
- Keep dynamic/DStab code paths out of the OpenDSS static comparison boundary.
- Use enabled static controls for acceptance comparisons.
- Use full result sampling only for bounded parity exports.
- Use monitor-only or streaming result output for production 8760 studies.
- Localize mismatches through device YPrim, KCL/KVL, load/generator terminal
  powers, capacitor/regulator state, and source-depth voltage plots before
  changing solver logic.
